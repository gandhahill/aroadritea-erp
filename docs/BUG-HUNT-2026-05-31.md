## Critical Bugs

### C-01: Reversed Journal Entry Status Never Updated — SPT Double-Counting

- **File**: `packages/services/src/accounting/reverse-journal.ts`, lines 181-197
- **Category**: Business rule violation
- **Module**: Accounting

**Root cause**: When reversing a journal entry, the code sets `reversedByJeId` on the original JE but never changes its `status` from `'posted'` to `'reversed'`. The schema supports the `'reversed'` status value, but the code never sets it.

**Impact**:
1. SPT Masa calculation (`spt-masa.ts` line 51) queries `eq(journalEntries.status, 'posted')` — reversed entries are still counted, **double-counting tax amounts**.
2. Trial balance, profit & loss, and balance sheet reports all include reversed entries.
3. Period closing logic counts posted JEs — reversed entries still count as posted.

**Suggested fix**:
```typescript
// In reverse-journal.ts, at the SET clause (lines 183-188):
// ADD status: 'reversed' alongside the existing reversedByJeId update
await db.update(journalEntries).set({
  reversedByJeId: reversalJeId,
  status: 'reversed',           // <-- ADD THIS
  updatedAt: new Date(),
  updatedByUserId: ctx.userId,
}).where(eq(journalEntries.id, originalJeId));
```

### C-02: Period Close Non-Transactional — Duplicate Closing Entries

- **File**: `packages/services/src/accounting/close-period.ts`
- **Category**: Race condition / data loss
- **Module**: Accounting

**Root cause**: The `closePeriod` function creates closing journal entries and updates the period status as separate non-transactional operations. If `postJournal` fails partway, the period can end up with partial closing entries, or if retried, duplicate closing entries. The code also silently discards `postJournal` failures — the return value is not checked.

**Impact**: Duplicate closing entries corrupt the general ledger. Period status can get stuck in an inconsistent state.

**Suggested fix**: Wrap the entire close-period operation in `db.transaction()`. Check the `postJournal` return value and rollback on failure. The bank-recon actions already use `db.transaction()` successfully, proving the driver supports it.

### C-03: Employee Delete/Hard-Delete Bypasses Permission System

- **File**: `packages/services/src/hr/delete-employee.ts`, line 26
- **File**: `packages/services/src/hr/hard-delete-employee.ts`, line 39
- **Category**: Permission bypass (CRITICAL SECURITY)
- **Module**: HR

**Root cause**: Both functions call `await requirePermission(ctx.userId, 'hr.employee.write')` but **silently discard the returned `Result<void>`**. The `requirePermission` function returns a Result type (does not throw on failure). Since the return value is never checked, the permission check is effectively a no-op.

**Compare with correct usage** in `createEmployee` (lines 38-41):
```typescript
const permCheck = await requirePermission(...);
if (!permCheck.ok) return permCheck;  // <-- correctly checks result
```

**Impact**: **Any authenticated user can delete or hard-delete any employee**, completely bypassing the permission system.

**Suggested fix** (apply to both files):
```typescript
const permCheck = await requirePermission(ctx.userId, 'hr.employee.write');
if (!permCheck.ok) return permCheck;

// Also fix the second permission check (location-scoped) in both files:
const locPermCheck = await requirePermission(ctx.userId, 'hr.employee.write', { locationId: existing.locationId });
if (!locPermCheck.ok) return locPermCheck;
```

### C-04: PB1 Tax Extraction Truncation — Systematic Overstatement

- **File**: `packages/services/src/pos/create-sale.ts`, lines 319-327
- **Category**: Money/price calculation
- **Module**: POS

**Root cause**: The `extractInclusiveTax` function computes `net = (inclusivePrice * 10000n) / (10000n + rateBps)`. BigInt division truncates (floors) the result. For PB1 at 10% (rateBps=1000), a price of Rp 25,000:
- `net = 250000000n / 11000n = 22727n` (truncated, true value is 22727.27...)
- `tax = 25000n - 22727n = 2273n`
- Correct tax should be 2272.72... → rounded to 2273 is acceptable, BUT the net is understated by 0.27

The identity `net + tax = inclusivePrice` holds, but the tax is **systematically overstated by ~1 rupiah per line** across all transactions. Over hundreds of daily transactions, this produces a measurable PB1 liability overstatement and revenue understatement.

**Suggested fix**: Compute tax first (truncate conservatively), then derive net as remainder:
```typescript
function extractInclusiveTax(inclusivePrice: bigint, rateBps: number): { net: bigint; tax: bigint } {
  const tax = (inclusivePrice * BigInt(rateBps)) / (10000n + BigInt(rateBps));
  const net = inclusivePrice - tax;
  return { net, tax };
}
```
This ensures tax is truncated (conservative for tax authority) and net gets the remainder.

### C-06: Donation Report Loads ALL Payments Globally

- **File**: `packages/services/src/reporting/donations.ts`, lines 91-99
- **Category**: Missing filter / performance bomb
- **Module**: Reporting

**Root cause**: The donation payments query has no `tenantId` filter and no `salesOrderId IN (...)` filter:
```typescript
// Fetches EVERY payment row in entire database with non-null donationAmount
db.select().from(payments).where(isNotNull(payments.donationAmount))
```
It then filters in JavaScript (`saleIds.includes(p.salesOrderId)`), loading all tenants' payment data into memory.

**Impact**: Full table scan on `payments`. Loads cross-tenant payment data into application memory. Will crash with OOM as data grows.

**Suggested fix**:
```typescript
db.select().from(payments).where(and(
  isNotNull(payments.donationAmount),
  inArray(payments.salesOrderId, saleIds)  // <-- ADD THIS
))
```

### C-07: COGS Report BOM Lines Query Has No WHERE Clause

- **File**: `packages/services/src/reporting/cogs.ts`, lines 112-122
- **Category**: Missing filter / performance bomb
- **Module**: Reporting

**Root cause**: The BOM lines query has no `.where()` at all:
```typescript
// Loads ENTIRE bom_lines table into memory
db.select(...).from(bomLines)  // no .where()!
```
Filtered in-memory on line 169 (`if (line.bomId !== bom.id) continue`). Similarly, ingredient products query (lines 127-137) fetches ALL products for the tenant.

**Impact**: Will cause OOM as BOM data grows. Full table scan on every COGS report generation.

**Suggested fix**:
```typescript
db.select(...).from(bomLines).where(inArray(bomLines.bomId, bomIds))
```
And for ingredient products:
```typescript
db.select(...).from(products).where(and(
  eq(products.tenantId, ctx.tenantId),
  inArray(products.id, ingredientIds)
))
```

### H-04: No Overlap Check for Accounting Period Date Ranges

- **File**: `packages/services/src/accounting/open-period.ts`, lines 55-66
- **Category**: Business rule violation
- **Module**: Accounting

**Root cause**: `openPeriod` checks for duplicate period codes but not overlapping date ranges. Two periods with different codes can overlap (e.g., "2026-05" covering May and "2026-05B" covering May 15 - June 15). Also, period code must match `YYYY-MM` format for `createJournal` to resolve correctly (it derives period code from `postingDate.substring(0, 7)`).

**Suggested fix**: Add overlap check query before insert:
```typescript
const overlap = await db.select().from(accountingPeriods).where(and(
  eq(accountingPeriods.tenantId, ctx.tenantId),
  lte(accountingPeriods.startDate, newEndDate),
  gte(accountingPeriods.endDate, newStartDate),
)).limit(1);
if (overlap.length > 0) return err('PERIOD_OVERLAP');
```

### H-16: Number Generator Race Condition — No Locking

- **Category**: Race condition
- **Module**: Shared

**Root cause**: All number generators (PO, invoice, journal, adjustment, opname, transfer) use SELECT MAX + 1 without any transaction or row locking. Two concurrent requests read the same MAX and generate duplicates. No unique constraint on `number` column to catch it.

## Cross-Cutting Themes

These recurring patterns should be addressed systematically, not just per-bug:

### 1. Missing `isNull(deletedAt)` (Soft Delete Leaks)
**Affected**: `list-employees`, `resolve-employee`, `update-employee`, `update-employee-login`, `list-my-schedule`, `disciplinary-service`
**Systemic fix**: Create a reusable query helper `withActiveTenant(table)` that always adds `isNull(table.deletedAt)` + `eq(table.tenantId, ctx.tenantId)`.

### 2. Non-Transactional Multi-Step Writes
**Affected**: `create-journal`, `delete-journal`, `close-period`, `create-po`, `outgoing-shipment`, `create-sale` (idempotency path)
**Systemic fix**: Audit all service functions that perform 2+ write operations. Wrap in `db.transaction()`. The Neon HTTP driver supports transactions (proven by bank-recon).

### 3. Permission Result Not Checked
**Affected**: `delete-employee`, `hard-delete-employee`
**Systemic fix**: Grep for `await requirePermission(` and verify every call site checks the returned Result. Consider changing `requirePermission` to throw on failure instead of returning Result, making it impossible to ignore.

### 4. Number Generator Race Conditions
**Affected**: All document types (PO, invoice, journal, adjustment, opname, transfer, helpdesk ticket)
**Systemic fix**: Implement a single `generateNextNumber(tx, tableName, prefix)` function that works inside a transaction with `FOR UPDATE`. Add UNIQUE constraints on all `number` columns.

### 5. Missing Tenant Scoping in Queries
**Affected**: `daily-summary.ts` (shifts), `donations.ts` (payments), `cogs.ts` (BOM lines), `coa/actions.ts` (taxRates update)
**Systemic fix**: Audit all queries for `tenantId` filter. Add a lint rule or query wrapper that enforces tenant scoping.

### 6. BigInt/Float Conversion for Money
**Affected**: `refund-sale.ts`, `cogs.ts`, `variance-service.ts`, `bank-recon/list-client.tsx`, `party-ledger-page.tsx`
**Systemic fix**: Create a `Money.scale()` utility that performs multiplication/division entirely in BigInt. Ban `Number(someBigInt)` in code review.

### 7. Missing Permission Checks on Server Actions
**Affected**: CMS reads, CRM member actions, promotions page data, notification settings
**Systemic fix**: Create a `withPermission(permCode)` higher-order function for server actions. Audit all server actions for permission checks.

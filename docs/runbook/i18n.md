# Runbook — i18n: adding & maintaining translation keys

The ERP is **Indonesian-first** with English and Mandarin parity.
All user-facing strings come from `next-intl` JSON dictionaries.

## Where the dictionaries live

| App   | Path                              |
| ----- | --------------------------------- |
| `web` | `apps/web/messages/{id,en,zh}.json` |
| `site`| `apps/site/messages/{id,en,zh}.json` |

Master data multi-language values live in DB as JSONB `{ id, en, zh }`
columns (products, categories, locations, etc.), not in these files.

## Adding a new key

1. Pick the **namespace** — usually the route segment (`pos`, `inventory`,
   `reporting`, etc.). Look at the existing JSON structure first; mirror
   the closest sibling.
2. Add the key to **id.json**, then **en.json**, then **zh.json**.
   All three must have it. Translation can be a stub in en/zh while you
   iterate; commit can wait until at least one stub is replaced.
3. In the component, call `useTranslations('pos').yourKey` or the
   server-side `getTranslations({ namespace: 'pos' })`.

## Naming conventions

- camelCase: `addProduct`, `outOfStock`, `customDonationLabel`.
- Group related keys: `donation.title`, `donation.confirm`,
  `donation.hint`. Use **dots** to nest if it improves grouping.
- Reuse generic verbs from `common.actions.*` (save, cancel, delete,
  edit) instead of duplicating per namespace.

## Locale-aware data

- `pickLocalized(jsonbCol, locale, fallback)` — reads `{ id, en, zh }`
  from a JSONB row.
- Numbers / currency: `new Intl.NumberFormat('id-ID')` everywhere.
  Money never uses `number` (precision); use `bigint` + format.
- Dates: ISO `YYYY-MM-DD` in DB, locale-aware in UI via
  `Intl.DateTimeFormat`.

## Parity check (quick)

```bash
node -e "
const id = require('./apps/web/messages/id.json');
const en = require('./apps/web/messages/en.json');
const zh = require('./apps/web/messages/zh.json');
function keys(o, prefix='') {
  const out = new Set();
  for (const [k,v] of Object.entries(o)) {
    const p = prefix ? prefix+'.'+k : k;
    if (typeof v === 'object' && !Array.isArray(v)) keys(v, p).forEach(s => out.add(s));
    else out.add(p);
  }
  return out;
}
const idK = keys(id), enK = keys(en), zhK = keys(zh);
const missingEn = [...idK].filter(k => !enK.has(k));
const missingZh = [...idK].filter(k => !zhK.has(k));
console.log('Missing in en:', missingEn);
console.log('Missing in zh:', missingZh);
"
```

Run this before merging. CI will eventually run the same check.

## Common gotchas

- **Forgetting to copy a key to all three locales** → next-intl will
  show the namespace-prefixed key in production for the missing locale.
- **Pluralization** → use `next-intl`'s `t('label', { count })` with
  `"{count, plural, =0 {Tidak ada} one {1 item} other {{count} item}}"`.
- **No HTML in JSON values** → next-intl renders as text. For rich
  formatting use rich-text helpers or pass `<strong>` via component.
- **Master data shadowing** → if a tenant overrides a category name in
  DB, the JSON dictionary's key for that category is irrelevant — UI
  reads from DB via `pickLocalized`.

## When to translate vs keep English

- **Brand product names**: keep English ("Bamboo Oolong Milk Tea")
  across all locales — they're brand identifiers, not phrases. See
  `packages/db/seed/recipes.ts` for an example.
- **Operational verbs**: translate ("Tambah" / "Add" / "添加").
- **Legal text**: Indonesian is the binding version; English and
  Mandarin are informational courtesy translations.
